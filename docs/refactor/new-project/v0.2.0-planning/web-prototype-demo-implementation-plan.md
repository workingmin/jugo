# Web 原型 Demo 实施计划

## 目标

本计划用于启动 `v0.2.0-planning` 需求评审阶段的 Web prototype demo 开发。原型目标不是复刻完整生产系统，而是用可交互页面验证新版本产品方向：

- 项目入口是否能清晰区分线性项目和多分支项目。
- 世界观、分支树、矩阵时序、节拍板是否能形成稳定的分层评审链路。
- 负责人是否能从全局结构逐层下钻到单场细节，并通过批注和校验报告完成需求评审。

## 工作目录

```text
prototype/v0.2.0-planning/
```

该目录作为新版本原型唯一开发入口。旧 `web-prototype/` 只作为 `v0.1.0-legacy` 参考资料，不在其上继续扩展新页面。

## 技术实施建议

| 项目 | 建议 |
| --- | --- |
| 原型形态 | 单页应用 SPA |
| 技术栈 | React + Vite |
| 数据来源 | 本地 mock 数据 |
| 后端依赖 | 无 |
| 存储 | 初期仅使用前端状态，可选 localStorage 保存演示状态 |
| 视觉风格 | 专业创作工作台，低饱和、信息密度适中、重点突出评审操作 |
| 全局主题 | 日间专业浅色模式 + 夜间创作深色模式，见 [web-prototype-theme-design.md](web-prototype-theme-design.md) |
| 主要适配 | PC 宽屏优先，兼容平板宽度；移动端仅保证不崩坏 |

## 页面与路由

| 页面 | 建议路由 | 是否首批必须 | 说明 |
| --- | --- | --- | --- |
| 项目管理首页 | `/` | 是 | 项目列表、空白态、新建项目模板选择 |
| 世界观知识库 | `/projects/:projectId/worldview` | 是 | 三栏结构，展示人物、势力、地点、规则、伏笔 |
| 分支树视图 | `/projects/:projectId/branch-tree` | 是 | 多分支项目专属，展示分叉、收敛、结局和任务分配 |
| 矩阵时序视图 | `/projects/:projectId/timeline` | 是 | 横向时间轴、纵向轨道、卡片、分支过滤 |
| 节拍板视图 | `/projects/:projectId/beat-board` | 是 | 单场细节、三幕分区、批注、AI 扩写入口 |
| 全局校验报告 | 弹窗 / 抽屉 | 是 | 不单独成页，从各视图顶部按钮触发 |

## 首批 Demo 范围

### 1. 项目管理首页

来源：[supporting/project-dashboard-design.md](supporting/project-dashboard-design.md)

必须展示：

- 有项目状态：项目卡片列表、项目类型标签、上次编辑时间、负责人、打开项目。
- 空白状态：新建项目引导、两套流程提示。
- 新建项目弹窗：线性单结局、多分支多结局模板选择。

首批可简化：

- 复制项目、删除项目、真实导出仅做按钮状态或提示。
- 团队项目切换先用静态标签表达。

### 2. 世界观知识库

来源：[supporting/worldview-knowledge-base-design.md](supporting/worldview-knowledge-base-design.md)

必须展示：

- 左侧分类：人物、势力、地点、世界规则、伏笔、道具。
- 中间条目列表：搜索、标签、条目预览。
- 右侧详情：人物档案字段、关联剧情、AI 上下文开关。
- 从关联剧情跳转到矩阵时序或节拍板的入口。

首批可简化：

- 富文本、图片上传、批量导入导出不做真实能力。
- 关系图谱不做。

### 3. 分支树视图

来源：[supporting/branch-tree-multi-ending-view-design.md](supporting/branch-tree-multi-ending-view-design.md)

必须展示：

- 主线公共节点、分叉选择节点、结局终点节点。
- 左侧分支层级和负责人分配。
- 右侧节点详情面板。
- 单分支筛选、跳转矩阵时序。
- 批注红点和冲突标记。

首批可简化：

- 节点拖拽可以先用静态布局加选中状态表达。
- AI 生成分支剧情仅做入口和模拟反馈。

### 4. 矩阵时序视图

来源：[supporting/matrix-timeline-view-design.md](supporting/matrix-timeline-view-design.md)

必须展示：

- 顶部视图切换、分支过滤、角色 / 势力过滤、全局校验。
- 左侧轨道管理：主线、角色线、反派线、势力线。
- 中央矩阵：横向时间刻度、纵向轨道、剧情卡片。
- 右侧卡片详情：关联人物、伏笔、批注、进入节拍板。
- 线性项目隐藏分支相关 UI；多分支项目显示分支标签。

首批可简化：

- 卡片拖拽先做视觉状态，不要求真实坐标持久化。
- 缩放和平移可先做控件展示或轻量 CSS 缩放。

### 5. 节拍板视图

来源：[supporting/beat-board-view-design.md](supporting/beat-board-view-design.md)

必须展示：

- 来源上下文：来自哪条分支、哪条轨道、哪个章节节点。
- 左侧幕结构：三幕 / 四幕区间。
- 中央节拍卡片：标题、梗概、人物、伏笔、批注红点。
- 右侧详情和批注面板。
- 返回矩阵时序、AI 扩写本场、校验设定冲突。

首批可简化：

- 场景全文生成仅做模拟结果。
- 批量复制至其他分支仅做按钮和提示。

### 6. 全局校验报告

必须展示：

- 设定冲突：人物行为违背人设。
- 伏笔问题：只埋不收或回收节点缺失。
- 时序问题：支线断层、时间顺序矛盾。
- 分支问题：结局体量失衡、分支条件缺失。
- 每条问题带定位入口：跳转世界观、分支树、矩阵时序或节拍板。

首批可简化：

- 校验结果全部来自 mock 数据，不做真实规则引擎。

## Mock 数据模型

首批建议维护一份可贯穿全部页面的 mock 数据，避免每个页面各自造数据导致联动断裂。

```text
Project
├── id
├── name
├── templateType: linear | branching
├── owner
├── updatedAt
├── worldEntries[]
├── branches[]
├── timelineTracks[]
├── timelineCards[]
├── beatCards[]
├── annotations[]
└── validationIssues[]
```

关键对象：

| 对象 | 用途 |
| --- | --- |
| `worldEntries` | 人物、势力、地点、规则、伏笔、道具 |
| `branches` | 多分支线路、分叉节点、结局节点、负责人 |
| `timelineTracks` | 矩阵时序纵向轨道 |
| `timelineCards` | 时序剧情卡片，关联世界观、分支和节拍 |
| `beatCards` | 单场节拍卡片，关联时序卡片 |
| `annotations` | 负责人批注，跨视图显示红点 |
| `validationIssues` | 全局校验报告 mock 问题 |

## 实施阶段

### 阶段 0：原型工程初始化

产出：

- 初始化 `prototype/v0.2.0-planning/` React + Vite 工程。
- 建立统一样式变量、基础布局和路由。
- 建立日间 / 夜间双主题 CSS variables 和主题切换入口。
- 建立 mock 数据文件。

验收：

- 本地 dev server 可启动。
- 首页可进入项目管理页面。
- 顶部导航和基础路由可用。
- 日间 / 夜间主题可切换，刷新后保留用户选择。

### 阶段 1：项目管理首页和模板选择

产出：

- 项目列表状态。
- 空白账号状态。
- 新建项目模板选择弹窗。
- 选择模板后进入对应项目工作台。

验收：

- 能演示线性项目和多分支项目两条入口。
- 线性项目后续页面自动隐藏分支树入口或分支控件。

### 阶段 2：统一工作台框架和全局校验抽屉

产出：

- 项目内统一顶部导航。
- 项目名称、模板类型、负责人身份标记。
- 全局校验报告抽屉。

验收：

- 所有项目内页面共享同一工作台框架。
- 校验报告可从任意核心视图打开，并能定位到对应页面。

### 阶段 3：世界观知识库

产出：

- 三栏知识库页面。
- 条目筛选、选中详情、AI 上下文开关。
- 关联剧情入口。

验收：

- 能从人物 / 伏笔条目跳到相关时序或节拍卡片。
- 能表达“世界观作为全项目底层约束”的产品逻辑。

### 阶段 4：矩阵时序和节拍板

产出：

- 矩阵时序轨道、时间轴、剧情卡片、详情面板。
- 节拍板三幕分区、节拍卡片、详情和批注面板。
- 从时序卡片进入节拍板、从节拍板返回时序。

验收：

- 能演示线性项目核心链路：世界观 -> 矩阵时序 -> 节拍板 -> 校验报告。

### 阶段 5：分支树和多分支联动

产出：

- 分支树画布、分支层级侧栏、节点详情。
- 单分支筛选。
- 从分支树跳转矩阵时序并携带分支上下文。

验收：

- 能演示多分支核心链路：世界观 -> 分支树 -> 单分支矩阵时序 -> 节拍板 -> 校验报告。

### 阶段 6：评审演示打磨

产出：

- 两条固定演示脚本。
- 关键按钮模拟反馈。
- 响应式和视觉细节整理。
- README 更新运行方式和评审路径。

验收：

- 需求评审人员无需解释代码即可按页面完成演示。
- 构建命令通过。
- 桌面宽屏下页面无明显遮挡、错位和信息溢出。

## 非目标

首批 demo 不做以下内容：

- 真实登录、权限和团队协作后端。
- 真实 API、数据库、WebSocket。
- 真实 AI 调用。
- 完整拖拽持久化和复杂画布引擎。
- 生产级导入导出。
- 移动端完整体验。

## 评审演示脚本

### 脚本 A：线性项目

1. 进入项目首页。
2. 新建线性单结局项目。
3. 进入世界观，查看主角、人设约束和伏笔。
4. 跳转矩阵时序，查看主线、角色线和反派线。
5. 选中异常卡片，进入节拍板。
6. 打开校验报告，定位人物行为冲突。

### 脚本 B：多分支项目

1. 进入项目首页。
2. 打开多分支互动项目。
3. 进入世界观，查看统一底层设定。
4. 进入分支树，筛选隐藏结局分支。
5. 跳转矩阵时序，只看该分支相关轨道卡片。
6. 进入节拍板，查看单场细节和批注。
7. 打开校验报告，定位伏笔未回收和分支体量失衡问题。

## 风险与处理

| 风险 | 处理 |
| --- | --- |
| 画布交互实现成本过高 | 首批用静态布局 + 选中状态 + 轻量筛选表达核心逻辑 |
| 页面数量多导致延期 | 先确保每页有可评审主路径，再补细节操作 |
| mock 数据不一致 | 使用单一数据源贯穿所有页面 |
| 旧原型干扰新设计 | 新 demo 只在 `prototype/v0.2.0-planning/` 开发 |
| 评审关注点分散 | 固定两条演示脚本，所有页面围绕脚本组织 |

## 完成标准

- `prototype/v0.2.0-planning/` 下存在可运行 Web 原型。
- 页面覆盖项目首页、世界观、分支树、矩阵时序、节拍板、校验报告。
- 两条评审链路可完整走通。
- mock 数据能在世界观、分支、时序、节拍和校验报告之间形成可见联动。
- 原型 README 记录安装、启动、构建和演示路径。
