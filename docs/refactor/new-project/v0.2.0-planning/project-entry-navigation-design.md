# 项目打开默认落点与历史位置策略

## 背景

`v0.2.0-planning` 的项目首页需要支持两类项目：

- 线性单结局：小说、影视剧本、短剧等线性叙事。
- 多分支多结局：互动小说、游戏叙事、多结局剧本。

用户从项目首页点击「打开项目」后，进入哪个页面会直接影响创作者的续写和负责人评审效率。

## 当前 Demo 策略

Web prototype demo 先不实现「上次打开位置」。为了让评审路径简单、稳定、可复现，项目打开时统一按项目模板进入默认落点：

| 项目类型 | 打开项目默认落点 | 世界观入口 |
| --- | --- | --- |
| 线性单结局 | 矩阵时序页面 | 顶部仅保留「世界观」「矩阵时序」两个主内容区 |
| 多分支多结局 | 分支树页面 | 顶部仅保留「世界观」「分支树」两个主内容区 |

这样可以优先验证核心产品判断：

- 线性项目的主要工作面是时序、章节推进、人物线和节奏。
- 多分支项目的主要工作面是分叉、选择条件、结局分布和分支体量。
- 世界观是底层约束，应当能随时进入，但不强制作为已有项目每次打开的第一页面。
- 项目编辑页顶栏不放产品 logo 和日间 / 夜间按钮，左侧只保留返回项目管理页面的「<」按钮和项目标题。
- 顶部中间的主内容区切换按钮按项目类型裁剪：线性项目只展示「世界观」「矩阵时序」，多分支项目只展示「世界观」「分支树」。
- 顶部右侧保留「保存」按钮，用于保存当前编辑更新；Web demo 阶段使用 localStorage 模拟草稿保存。
- 项目工作台顶栏已有「世界观」Tab，不再额外放置顶栏级「编辑世界观」按钮，避免重复入口干扰主操作。

## 为什么暂不做「上次打开位置」

「上次打开位置」在个人单机工具里很自然，但在多人团队协作系统里需要额外设计，否则容易产生失效和误导。

### 问题 1：记录维度必须是个人用户

项目是多人共同创作。甲用户和乙用户的关注点可能不同：

- 负责人上次打开分支树做结构评审。
- 编剧上次打开某个节拍卡片写单场。
- 世界观编辑上次打开人物档案。

因此「上次打开位置」不能记录在项目全局维度，只能记录在：

```text
user_id + project_id
```

必要时还要包含：

```text
view_type + node_id + filter_state + scroll_position
```

### 问题 2：历史位置可能被其他用户删除或改写

多人编辑时，甲用户上次打开的页面节点可能被乙用户删除、合并、移动或权限收回。例如：

- 分支节点被删除。
- 时序卡片被合并。
- 节拍卡片被拆分。
- 世界观条目被归档。
- 当前用户不再拥有该分支权限。

这会导致历史位置失效。

### 问题 3：失效后必须回退默认落点

如果系统后续实现「上次打开位置」，每次进入项目都需要先校验历史位置是否仍有效：

1. 当前用户是否仍有访问权限。
2. 目标视图是否仍对该项目类型开放。
3. 目标节点是否仍存在。
4. 目标节点是否未归档 / 未删除。
5. 相关筛选条件是否仍可用。

任一校验失败，都应回退到项目类型默认落点：

- 线性单结局：矩阵时序。
- 多分支多结局：分支树。

## 后续正式产品建议

正式产品可以在默认落点基础上增加个人历史位置，但必须作为增强能力，而不是进入项目的唯一依据。

建议优先级：

1. MVP / Web demo：只做项目类型默认落点。
2. Alpha 阶段：记录个人用户维度的最近视图。
3. Beta 阶段：记录个人用户维度的最近节点，并做有效性校验。
4. 协作稳定后：支持「回到上次位置」和「进入默认工作台」两个入口。

## 当前验收标准

- 从项目首页打开线性单结局项目，进入矩阵时序页面。
- 从项目首页打开多分支多结局项目，进入分支树页面。
- 线性单结局项目编辑页顶部只展示「世界观」「矩阵时序」。
- 多分支多结局项目编辑页顶部只展示「世界观」「分支树」。
- 两类目标页面都保留世界观入口；顶栏使用「世界观」Tab，页面内容区可保留上下文型「编辑世界观」面板入口。
- 项目编辑页右上角展示「保存」按钮，点击后能反馈保存状态。
- 不记录、不读取、不展示「上次打开位置」。
