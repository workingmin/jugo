# 版本策略

## 背景

当前 JUGO 项目尚未正式上线，但需求发生大幅变化，后续重构不再是单纯改代码，而是从需求分析开始重新立项。因此需要把旧项目资料固定为一个可追溯版本，同时给新项目建立独立的规划版本。

## 版本命名

| 版本 | 类型 | 说明 |
| --- | --- | --- |
| `v0.1.0-legacy` | 旧项目归档版 | 保存当前未上线实现、方案稿、原型、契约差异和现状盘点 |
| `v0.2.0-planning` | 新项目规划版 | 重构后项目的立项和需求分析起点 |

## 为什么不使用 `v1.0.0`

`v1.0.0` 应保留给首个正式可发布版本。当前旧项目未上线，新项目还处于需求分析阶段，因此继续使用 `0.x` 预发布版本更准确。

## 为什么新项目用 `v0.2.0-planning`

JUGO 的产品名称和仓库历史仍然延续，但需求和实现要重新开始。使用 `v0.2.0-planning` 可以表达两层含义：

- 它不是旧实现的补丁，所以不使用 `v0.1.x`。
- 它还不是可上线版本，所以不使用 `v1.0.0`。

## 归档规则

- `v0.1.0-legacy` 归档后不再承接新需求。
- 旧资料只允许做勘误、补充索引和来源说明，不做功能性改写。
- 新需求、重新设计、技术选型和架构决策全部进入 `v0.2.0-planning`。
- 如果后续产生可运行的新版本，建议从 `v0.3.0-alpha.0` 或 `v0.3.0` 继续推进。

## Git 标记建议

在本次文档归档提交完成后，建议给仓库打一个不可混淆的标签：

```bash
git tag jugo-v0.1.0-legacy
```

如果希望新立项也固定起点，可在新需求文档初版完成后再打：

```bash
git tag jugo-v0.2.0-planning
```

标签应在确认工作区干净、归档文档已提交后执行。
